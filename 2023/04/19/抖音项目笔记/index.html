微服务

微服务优势

  1. 各个微服务可以各自开发，并专注于单一的功能，对开发人员友好
  2. 容易扩展、更改
  3. 各服务彼此独立，能够做到负载均衡

微服务缺点

  1. RPC增加延迟
  2. 增加技术故障排除的挑战

数据库部分

需求分析

  • 用户信息表：用户名，密码，关注人数，粉丝人数，作品数，点赞数
  • 用户关系表：用户之间的关注信息
  • 视频信息表：作者id，视频标题，视频和封面的url，点赞数，评论数
  • 点赞信息表：用户id，视频id
  • 评论信息表：评论id，视频id，内容，时间
  • 消息信息表：消息发送方id，消息接收方id，消息内容，时间

注意点

  1. 使用了gorm.Model，因此会自动有id、创建、删除和更新的时间。gorm中自动实现了软删除，通过判断删除时间字段是否填充来标记是否被删除，如果为空说明没有被删，否则代表已被删，并且其值就是删除时间。
  2. 为了避免某些已删除的数据与新插入的数据之间的冲突，必须在某些表的唯一索引中加入软删除字段。例如favorite 和 follow，如果不加入软删除，那么在插入新的数据时，会和原有数据冲突，比如取关后再点关注；而comment等不会产生这个问题，因为同一个用户同一个视频可以存在多个评论。
  3. relation和favorite为了避免重复插入，不再使用gorm.Model提供的delete_at，而是使用soft_delete.DeleteAt，将默认值从null变为0。
  4. 为了保持comment和message在redis和mysql的一致性，需要生成数据的uid，因此添加了snowflake雪花算法生成的uid以及自己设置的时间字段。

Redis部分

需求分析

  • 用户信息：key为用户id，value分为两部分，一部分是用户的固定信息，包括id和用户名等；另一部分是统计信息，包括关注数、粉丝数、作品数等。统计信息经常被修改。
  • 视频信息：key为视频id，value分为两部分，一部分是视频的固定信息，包括视频id，作者id，标题，视频和封面url等；另一部分是统计信息，包括关注数、获赞数等。统计信息经常被修改。
  • 用户关系：即关注列表和粉丝列表。key为用户id，value为用户id的数组。在这里redis的数据结构选用set，这样可以对关注列表和粉丝列表求交集。
  • 点赞列表：key为用户id，value为视频id的数组。
  • 投稿列表：key为用户id，value为视频id的数组。
  • 评论信息和消息信息，评论列表和消息列表也都存入缓存中。

注意点

  1. 缓存穿透：用户访问的数据既不在缓存在，也不在数据库中，导致数据库压力剧增。

    解决：使用布隆过滤器来快速判断是否存在该id，避免直接通过查询数据库。当写入用户信息时，就添加用户id到布隆过滤器中。

    布隆过滤器：基于若干哈希函数和二进制位数组来实现。当访问一个数据时，先通过若干次哈希将其映射为一个二进制数，并检查这个二进制数是否存在。可以知道的是：当二进制数中任意一位为0，则说明数据库中一定不存在该数据；当二进制全部都为1时，也只能说明可能存在该数据，而不能说明一定存在；由于哈希函数不可逆，所以无法通过二进制来反向推导元素。

  2. 缓存雪崩：大量缓存数据同时过期，造成大量请求直接访问数据库，导致数据库压力剧增。
    解决：过期时间加入随机数，如一天+随机0-60分钟，是缓存失效时间尽量均匀，防止雪崩。

  3. 缓存击穿：热点数据过期，导致大量请求直接访问数据库。

    解决：分布式互斥锁。

    为什么要采用分布式互斥锁？

    假设某个大V粉丝列表缓存过期，但他此时还有粉丝正在关注，就会同时有很多请求访问数据库，如果使用分布式锁，来使得第一个拿到锁的用户来数据库拿到表，并同时将其缓存到redis中，其他用户等待解锁后直接在缓存中取出。

    如何实现：使用redis的set nx ex命令来加锁并设置过期时间，并在解锁的时候校验锁是否已被释放，在此过程中使用lua脚本保证原子性。增加看门狗机制，在加锁成功后不断延迟锁的过期时间，释放锁或报错时关闭看门狗。

  4. Lua脚本：redis采用相同的解释器来运行所有命令，来保证脚本的原子性。

为什么不用etcd

etcd也是键值存储系统，一般用于服务发现，支持服务注册、服务健康检查等等，但其读写延迟和吞吐量上都不如redis，并且也没有缓存过期策略等等，而redis性能较高，拥有主从复制、缓存过期策略等优势。

Pulsar

需求分析

针对可能的高频写入操作场景，会先生产消息到消息队列，然后发送消息成功再写入redis，并异步返回，通过服务消费pulsar消息，再写入mysql，以进行削峰，异步操作还能降低点赞等操作的耗时，改善用户体验，并实现redis和mysql的最终一致性。

  • 用户服务：关注/取关
  • 视频服务：点赞/取消点赞
  • 评论服务：评论/删除
  • 消息服务：消息存储

为什么选择pulsar

将生产和消费解耦，限流削峰。高并发场景下的性能瓶颈在于mysql的读写，经过消息队列可以缓解mysql的压力。

  • rabbitmq：容易部署和使用，但消息堆积时性能急剧下降，性能不高。
  • kafka：吞吐量高，但有延迟
  • pulsar：延迟低，吞吐量可以，消息处理和存储是分开的，可以无缝修复问题、复制等。

实现

每个微服务都建立了一个pulsar客户端，使用一个消费者协程一直阻塞等待生产，当关注等操作发生时，就会生产消息，而消费者接收到消息后调用ACK表示成功，之后再将schema消息反序列化为消息体消息写入mysql，如果失败则会删除redis中的key保证一致性。

链路追踪

在微服务架构中，请求在各服务之间流转，调用链错综复杂，为了定位问题和异常，需要链路追踪。链路追踪可以追踪并记录请求在系统中的调用顺序、调用时间等，可以帮我们定位异常服务和性能瓶颈。

在项目中使用了jaeger，我们使用context传递span。

Context

context 包含了请求的上下文信息，例如请求的截止时间、请求的用户信息、请求的语言环境等等。context 的主要作用包括：

  1. 控制请求的生命周期，而避免使用通道+select
  2. 传递请求信息，包括用户信息、语言环境等
  3. 控制并发访问，可以多个goroutine共享一个context，控制访问的权限和并发度
  4. 中间件，可以在context中保存请求的状态和信息，在中间件进行处理，然后传递结果

安全设计

nginx限流

基于漏桶的思想实现限流，即缓存请求、匀速处理，多余的请求直接被丢弃。

令牌桶算法：在固定的时间间隔内，按照一定速率向令牌桶中添加令牌，当请求到达时，如果令牌桶中有足够的令牌，则可以处理请求并从令牌桶中取走一个令牌，否则就拒绝请求。可以控制流量的平均速率和峰值速率。

加密算法

采用argon2id算法并随机加盐再哈希。

为什么argon2id：快、安全。

加盐：在hash之前先将hash内容中固定位置插入字符串 使得hash中加入字符串后的结果与不加时不同 增加安全性。

存储：将argon版本等参数和密码哈希、盐哈希共同生成一个字符串并存储起来，在登录时比较输入密码生成的哈希字符串是否和原字符串相同。

登录逻辑：比较密码哈希，一致则直接通过，不一致开始记录失败，如果十分钟内超过5次则冻结登录30分钟。

输出控制：不在日志中打印和保存密码、秘钥等，提示错误时也只提示用户名/密码错误而不打印明文。
